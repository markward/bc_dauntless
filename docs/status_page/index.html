01 The Experiment

What this is, and what it isn't

This is an experiment, and we don't know how it ends. The goal is simple: take Star Trek: Bridge Commander — a 2002 Windows-only game — and rebuild its closed-source engine from scratch, so the original game scripts run again on modern macOS, Linux, and Windows.

Its original team was ambitious, pushing Python scripting to capture the scale of Star Trek. 24 years on, we're being just as ambitious — bringing those scripts back to life with AI.

That's what makes this unusual. Almost every line of code here was written with Claude (Anthropic's LLM) as a pair-programmer: the NIF asset loader, the C++ renderer, the host loop that embeds CPython, the physics, the shaders — all of it. As far as we know, it's the first serious attempt to rebuild a legacy game engine this big with an LLM as the principal collaborator.

It might not work. It could stall at Combat MVP, with the AI scripts running but combat never feeling right. It could hit a wall at bridge characters. Or it might surprise us and land. Either way, this page is here to make the attempt visible — what's built, what's next, and what broke last week.

02 Mission

What we're building

We're building a modernised version of the Bridge Commander game engine, with Claude Code as our pair-programmer. The aim is simple: take a beloved 2002 game that only ever ran on Windows, and bring it back to life on the machines people actually use today — without losing what made it special.

The original team left behind a remarkable gift. Every mission, every ship, every line of dialogue lives in plain, readable scripts they released as an SDK. The game logic still works. What's missing is the engine underneath — the part that draws the ships, plays the sounds, and ties it all together. That's what we're rebuilding.

Faithful to the original

The original missions, ships, and scripts run as-is. We're rebuilding the engine, not the game — every mission you remember should play as they are.

Modern visuals

A brand-new graphics engine, built from scratch, drawing the same ships and effects but with the required upgrades to look even better on today's displays.

Runs everywhere

macOS first, Linux next, Windows for the 1.0 release. No emulators, no compatibility layers — for the first time, Bridge Commander will run natively beyond Windows.

Open and community-friendly

Every commit is public. The fans who kept Bridge Commander alive for two decades are the natural home for this — and mod support is firmly on the roadmap.

03 Latest Signal
Stardate 2026-06-10 · Here comes the fun part

The original Python game files keep us anchored. They remain Claude's best source of truth — though as we push into the more intricate corners of the game, I'm increasingly having to insist on a deep read of the source before guessing at how something is meant to work.

In under a month we've mapped enough of the game to stand up the fundamentals of combat. Careful measurement has the ships' speed and movement matching the original, and paired with STBC-faithful camera controls, it feels very familiar indeed.

Units, scaling and coordinate systems have been the real pain these past two weeks though. Claude thankfully caught that it has been inventing inconsistent coordinate spaces, and fixed that just in time. It still struggles to infer what "the right size" or "the right angle" should be from the original Python alone. We're nearly through it — the stragglers are star-system coordinates (planets loading on top of moons) and phaser arrays I suspect are firing in reverse.

But now we're at the fun part. More than once I've sat down to debug a problem and got happily sidetracked tearing apart an enemy Galaxy-class ship — all the more fun now I've allowed time for some side quests to add VFX I've been dreaming about for years.

Getting ship combat working end to end, then on to the Dauntless reinterpretation of Quick Battle to round it all out.

05 Under the Hood

Technical stack

Host process
C++ executable embedding CPython 3 — drives the BC Python game loop natively, no shim DLL.
Renderer
In-house C++ renderer parsing BC's NIF asset blocks. Passes: opaque hull, sovereign shields, glow (via AddLOD), space dust.
Physics
PyBullet during the early headless work; transitioning to native Bullet through the C++ host.
Audio
OpenAL — slated to stand up alongside the audio script bindings.
UI
Chromium Embedded Framework (CEF) — menus and HUD overlays are built as HTML/CSS/JS, composited over the renderer; a component library mirrors the original BC LoadInterface scripts.
Asset pipeline
BC NIF files (NetImmerse / Gamebryo blocks). Right-handed coordinate space, W-first quaternions, link-ID streaming.
Platforms
macOS-first development. Linux follows. Windows joins for 1.0. Single codebase, no DirectX dependency.
Source material
BC SDK v1.1 — 1,228 Python files, fully readable, unchanged since 2002.

How Claude fits

The collaboration model in plain English: a human (Mark) directs, reviews, and makes architecture calls. Claude writes the bulk of the C++, shaders, build scripts, and tooling. Each non-trivial change goes through a brainstorm → spec → plan → implementation loop, with the spec committed to docs/superpowers/specs/ before code is written. Every commit is human-reviewed before merge.

The repository is structured to make this auditable: design specs are first-class artefacts, surveys of the original game's behaviour live alongside the code, and the project's CLAUDE.md is itself documentation for both human and AI readers.
