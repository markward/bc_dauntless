Sensors panel — perspective radar, engine-rendered

RadarDisplay extends STStylizedWindow (App.py:8513), so our locked panel chrome wraps it. The disc and blips are engine-rendered — the Python shim creates the widget, registers it with the tactical window, and exposes the colour palette; the C++ renderer paints the perspective disc and animates blips per frame. The visual shifts from flat-top-down to a horizontal disc viewed at perspective, so blip altitude reads as literal screen height.

Encoding (perspective disc)

Disc anchor (small dot on disc) = horizontal-plane position.
Vertical stem = altitude — up from disc = above, down = below, length = magnitude.
Triangle = the contact itself, at the stem's far end, rotated to indicate horizontal heading.
Colour = affiliation (blue friendly, red hostile, gold neutral). The disc itself shows non-linear range falloff via concentric range rings.

SDK alignment

  • RadarDisplay_Create(w, h) at App.py:8513 — Python factory call. Inherits STStylizedWindow, so panel chrome is automatic.
  • SetUseScrolling(0) — radar disc never scrolls (called at TacticalMenuHandlers.py:474).
  • pTacticalWindow.SetRadarDisplay(pRadarDisplay) — registers the widget so the C++ TacticalControlWindow updates blip data every frame.
  • RadarBlip extends TGIcon (App.py:8535) — each contact is its own positioned icon child managed by the C++ renderer.
  • Python's role: create the widget, wire it into the tactical window, expose the colour palette. C++ engine's role: query contacts, project them onto the tilted disc plane, compute stem heights from world Z-offset, paint each blip per frame.
  • The perspective tilt is a pure renderer choice — no SDK signature changes. A future option/preference toggle could swap between flat-top-down and perspective views without touching the Python side.

Open question

The spec mentions a torpedo type-specific icon ("differentiated by torpedo type so threats can be assessed"). For now I'm using a small ▶ chevron in neutral gold. We have two practical options when we get to the C++ renderer:

  1. Per-type small textures (photon / quantum / plasma / etc.) — most authentic, requires asset work.
  2. Single chevron, with type indicated by colour (photon = blue-ish, quantum = purple, plasma = orange). Requires no assets, less informative.

Decision can wait until the C++ renderer is up; the visual chrome doesn't need to lock the per-type icons today.
