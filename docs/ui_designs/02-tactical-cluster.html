What's locked here

  • Topology: 1 panel at top-left (Tactical), 3-panel cluster at top-right (Orders, Manoeuvres, Tactics). Matches the four STStylizedWindow_CreateW calls at TacticalMenuHandlers.py:346, 592, 668, 681.
  • Orders 2×2: Destroy / Disable / Stop / Evade arranged in two rows of two — matches Python's explicit AddChild(pButton, fXPos, fYPos, 0) positioning at lines 607–619.
  • Manoeuvres / Tactics compact state: single-row "▸ <chosen>" with a ▼ expand affordance. This is the STCharacterMenu in its Close()'d state — same widget that, when opened, expands into the full option list.
  • Tactics open layout: 3×3 compass with Top/Bottom on a paired vertical axis. Inside an STMenu, Python only calls AddChild(pButton) with no position (CreateOrderMenu at line 452), so layout is ours to choose.

Visual vocabulary recap

  • Header gradients: panel's primary SDK colour → +20° hue. Tactical & Orders use g_kSTMenu1NormalBase rgb(216,94,86); Manoeuvres & Tactics use g_kSTMenu3NormalBase rgb(207,96,159).
  • Chosen / action rows: g_kSTMenu2NormalBase purple at 22% alpha bg + lighter rgb(173,132,255) left accent + bright text.
  • Off / inactive rows: dim grey rgba(40,40,55,0.5) bg + rgb(120,120,140,0.5) accent + text rgb(180,180,200).
  • Toggle bullet: filled gold rgb(255,210,90) with soft glow when ON; hollow purple-edged when OFF.
  • Chosen caret ▸ + expand chevron ▼: gold rgb(255,210,90), drawing the eye to current value + clickable affordance.

Open question

Manoeuvres has 4 options. When opened, should it use the same 2×2 grid as Orders (consistency with the other STMenu1 panel) or a vertical stack (consistency with how dropdowns usually expand)? I lean toward vertical stack — Manoeuvres is semantically a list of "how to engage" not a grid of distinct choices, and the 2×2 only makes sense in Orders because Python forces it there. Easy to swap if you disagree.
